City Simulation

A systems-focused simulation project built in a custom C++ engine, exploring how NPCs react to needs, schedules, scarcity, and changing world state instead of following fixed scripts.

Gameplay / AI / Systems Programmer

Simulation Systems, AI, World Interaction

Overview

The goal of this project was to create a simulation where NPCs feel alive by reacting to the world and their own needs instead of following a script.

Rather than focusing on a single mechanic, I wanted to build a small systemic world where actors move through daily routines, respond to scarcity, and change their behaviour depending on what is actually happening around them.

Simulation & Actor Behaviour

A typical actor moves through a simple daily loop — working, getting hungry, eating, getting tired, relaxing, and eventually going to sleep. What drives this behaviour is not a fixed sequence, but a combination of needs and what is currently available in the world.

Hunger is the strongest need, but actors still depend on what exists in the world. Without reservation, this can lead to weird behaviour.

For example, multiple NPCs can decide to go to the same restaurant at the same time, even if it will run out of food before they arrive. When that happens, they cancel their action and switch to something else. If food suddenly becomes available again, they may all go back at once, which can make the behaviour look unstable.

AI Structure – Brain & Actions

I built the AI around a simple Brain + Action structure.

The Brain decides what an actor should do based on needs, schedule, and the current world state, while Actions handle how that behaviour is executed.

Each frame, the actor either continues its current action or switches to a new one if the action has finished or should be interrupted. Some actions, like eating, sleeping, building, or carrying resources, are not allowed to be interrupted once they have started.

This setup stayed very simple, but that was also its biggest strength. It made the system easy to reason about, easy to debug, and easy to extend with new behaviours and rules.

One important detail was making sure certain actions were marked as non-interruptible. If I forgot to do that, newly added actions could instantly cancel themselves, which made bugs very obvious and helped me quickly understand what was going wrong.

At the core of the AI is a very simple update loop:

View Brain::Update pseudocode
BRAIN::UPDATE()
if there is no current action
or the current action is finished
or the current action should be interrupted
{
    decide next action
    create that action

    if a new action exists
    {
        start the action
    }
}

if there is a current action
{
    update the action
}

From Static to Living Systems

One of the biggest challenges in this project was making the simulation actually feel alive.

Early on, the behaviour looked correct on a technical level, but it felt static. Actors repeated the same patterns over and over again, there were no real consequences to their decisions, and too little changed in the world.

The biggest improvement came from introducing finite resources.

Once resources could run out, actors were forced to react to each other and to the world in a different way. Suddenly, behaviour became less predictable — actors could no longer rely on things always being available, which made their decisions feel more grounded.

I also introduced food decay, partly because I thought it would be fun to implement, but also because it added another layer of scarcity. Food is no longer just available or unavailable — it changes over time. This created more variation in the simulation and added subtle pressure to the resource and transport systems.

Together, these changes made the world feel less perfect, and in turn, more believable.

Reservation & Shared World State

Reservation became an important part of making the simulation more stable and believable.

Some tasks need to be claimed before the actor actually starts doing them. In my current project, this matters most for carrying resources and for build sites, where a maximum of two actors can work on the same construction at once.

Without this, too many actors could commit to the same task, which created messy and inconsistent behaviour. Adding reservation helped reduce that problem and made task distribution feel much more intentional.

Balancing & Iteration

Balancing the simulation turned out to be one of the hardest parts of the project.

Small changes to values could completely change how the actors behaved. At different points, NPCs would eat too often, go to sleep too early, or behave in ways that didn’t feel natural in the context of the world.

What made this challenging was how connected everything is. Needs, schedules, available resources, and action rules all influence each other, which made it difficult to predict the outcome of even small adjustments.

This led to a lot of iteration and constant tweaking of thresholds and values. It was sometimes frustrating, but also one of the most valuable parts of the project.

It made me understand how sensitive simulation systems are, and how important it is to design them in a way that makes them easier to observe, debug, and adjust.

What I’d Improve Next

The next step would be to push the simulation further by adding more direct interaction between actors, improving reservation around restaurants, and making the world state create even more meaningful variation over time.

I would also like to improve pathfinding robustness and continue expanding the debug tools, since that became a huge part of making this kind of project readable and manageable.
